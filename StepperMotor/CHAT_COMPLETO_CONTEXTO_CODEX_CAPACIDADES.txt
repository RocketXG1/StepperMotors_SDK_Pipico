================================================================================
1. OBJETIVO GENERAL DEL PROYECTO
================================================================================

El proyecto busca crear un template inicial en Pico SDK C/C++ para controlar una máquina cartesiana con Raspberry Pi Pico / RP2040. El sistema está diseñado para controlar de 1 a 8 ejes con motores paso a paso, usando:

- Pico SDK C/C++.
- PIO para generar pulsos STEP.
- GPIO normal para DIR, ENABLE, HOME y límites.
- Perfil trapezoidal de velocidad.
- Sincronización de ejes con algoritmo tipo Bresenham.
- Movimiento lineal directo.
- Cola de movimientos.
- Suavizado de trayectoria en esquinas por medio de MotionPlanner.
- HOME físico por eje.
- HOME simulado para pruebas sin sensores.
- Orden de HOME configurable.
- Orden de HOME por nombre, no solo por índice.
- Nombres obligatorios de ejes.
- Validación de nombres duplicados.
- Conversión automática de nombres de eje a MAYÚSCULAS.
- Límites físicos mínimo y máximo.
- Límites por software mínimo y máximo.
- Movimiento seguro después de HOME.

El template actual instancia 3 motores/ejes:

- X
- Y
- Z

pero la arquitectura permite mínimo 1 eje y máximo 8 ejes.

4.3 StepperPioPulseEngine.h / StepperPioPulseEngine.cpp
-------------------------------------------------------
Clase encargada de enviar máscaras STEP al PIO.

Responsabilidades:
- Cargar el programa PIO.
- Inicializar la state machine.
- Manejar un grupo de 8 pines STEP consecutivos.
- Validar que un pin STEP pertenezca al grupo.
- Convertir GPIO STEP a offset de máscara.
- Enviar máscara STEP bloqueante o no bloqueante.

Parámetros de instancia:
StepperPioPulseEngine pulseEngine(
    pio0,
    0,
    2,
    10
);

Significado:
- selectedPioBlock = pio0.
- selectedStateMachine = 0.
- firstStepGpio = 2.
- minStepIntervalUs = 10.

Con firstStepGpio = 2 y STEP_GROUP_PIN_COUNT = 8:
- GPIO2 = STEP offset 0.
- GPIO3 = STEP offset 1.
- GPIO4 = STEP offset 2.
- GPIO5..GPIO9 disponibles para otros ejes STEP.

4.4 stepper.pio
---------------
Programa PIO que genera pulsos STEP.

Diseño:
Cada dato recibido por FIFO es una máscara de 8 bits.

Ejemplo:
- bit 0 = STEP eje con pin offset 0.
- bit 1 = STEP eje con pin offset 1.
- bit 2 = STEP eje con pin offset 2.

El programa:
- pull block
- out pins, 8
- mov pins, null

Esto pone en alto los pines indicados por máscara y luego los baja.

Importante:
Los STEP deben ser pines consecutivos para este diseño PIO.

5. DECISIONES IMPORTANTES TOMADAS EN EL CHAT

5.1 HOME secuencial, no paralelo
--------------------------------
homeAll() hace HOME en secuencia.

Antes estaba fijo como:
- Z
- X
- Y

Después se cambió para que el orden ya no estuviera fijo dentro de homeAll().

5.2 HOME configurable por función
---------------------------------
Se agregó:
setHomeOrder({2, 0, 1})

Esto permite definir el orden por índice.

5.3 HOME configurable por nombre
--------------------------------
Se agregó:
setHomeOrderByName({"Z", "X", "Y"})

Esto permite definir el orden por nombre del eje, lo cual es más visual y evita confusión con índices.

Ejemplo:
Si:
axes[0] = X
axes[1] = Y
axes[2] = Z

Entonces:
setHomeOrderByName({"Z", "X", "Y"})

Internamente encuentra:
- Z -> índice 2
- X -> índice 0
- Y -> índice 1

Y ejecuta:
- homeAxis(2)
- homeAxis(0)
- homeAxis(1)

5.4 Validación de nombres
-------------------------
Se pidió:
- Evitar nombres duplicados.
- Convertir nombres a mayúsculas.
- Hacer obligatorio reemplazar "UNNAMED".

Se implementó:
- config.setAxisName("X")
- config.setAxisName("Y")
- config.setAxisName("Z")

Si algún eje queda "UNNAMED", motion.isValid() devuelve false.

Si hay nombres repetidos, motion.isValid() devuelve false.

5.5 Movimiento directo vs cola
------------------------------
Se aclaró:

moveLinearTo(...)
- Movimiento directo.
- No usa MotionPlanner.
- No aplica suavizado geométrico.
- Sí usa perfil trapezoidal, límites, sincronización y Bresenham.

queueLinearTo(...) + prepareQueuedMoves()
- Usa MotionPlanner.
- Puede aplicar suavizado si configurePathSmoothing(true, ...) fue llamado.
- Recomendado para múltiples puntos y trayectoria con esquinas.

5.6 Sincronización de motores
-----------------------------
Se aclaró que MotionPlanner NO es quien garantiza que todos los motores lleguen al mismo tiempo.

El encargado es AdvanceMotionController mediante:
- buildLinearBlockToArray()
- dominantSteps
- bresenhamError[]
- executeBresenhamStep()

Ejemplo:
Si:
- X necesita 8000 pasos
- Y necesita 4000 pasos
- Z necesita 1000 pasos

Entonces:
- dominantSteps = 8000
- X se mueve casi cada evento.
- Y se mueve proporcionalmente.
- Z se mueve proporcionalmente.
- Todos llegan cuando termina el movimiento dominante.

5.7 emergencyStop() y tight_loop_contents()
-------------------------------------------
Se explicó:
tight_loop_contents() es una función del Pico SDK usada dentro de ciclos infinitos intencionales.

Ejemplo:
while (true) {
    tight_loop_contents();
}

Significa:
"El programa se queda aquí detenido intencionalmente."

No reinicia la Pico.
No reintenta HOME.
No mueve motores.

Se habló de que para un sistema real conviene implementar estado de falla y botón de reset/reintento, pero en el template se dejó como paro básico para prueba.

8. HOME
================================================================================

homeAxis(axisIndex)
-------------------
Ejecuta HOME de un solo eje.

Secuencia:
1. Valida controlador.
2. Valida eje.
3. Verifica si HOME está habilitado.
4. Habilita eje.
5. Si HOME simulado está activo y sensor simulado activo:
   - asigna posición HOME.
   - marca homed true.
   - termina.
6. Define dirección HOME con getHomeDirectionPositive().
7. Calcula delays de seek y latch.
8. Convierte maxHomingTravelMm a pasos.
9. Busca HOME rápido.
10. Si excede recorrido máximo, emergencyStop().
11. Hace backoff.
12. Espera 100 ms.
13. Entra lento al sensor.
14. Asigna posición lógica HOME.
15. setHomed(true).

homeAll()
---------
Ejecuta HOME de todos los ejes configurados en homeOrder.

No tiene orden fijo.
Requiere llamar antes:
setHomeOrderByName({"Z", "X", "Y"})

o:
setHomeOrder({2, 0, 1})

Es secuencial:
- termina un eje
- luego pasa al siguiente

- Respeta límites físicos.
- Respeta límites por software.
- Respeta velocidad máxima por eje.
- Respeta aceleración máxima por eje.
- Respeta frecuencia máxima de StepperPioPulseEngine.

Uso:

while (motion.isBusy()) {
    motion.update();
}

================================================================================
11. PIO Y STEP
================================================================================

El PIO recibe una máscara.

Ejemplo:
Si se quiere mover X, con STEP GPIO 2 y firstStepGpio = 2:
offset = 0
mask = 1 << 0

Si se quiere mover Y:
GPIO 3
offset = 1
mask = 1 << 1

Si se quiere mover X y Y al mismo tiempo:
mask = (1 << 0) | (1 << 1)

StepperPioPulseEngine manda la máscara a la FIFO:
pio_sm_put_blocking(...)

stepper.pio pone esos pines en alto y luego en bajo.

================================================================================
12. SINCRONIZACIÓN BRESENHAM
================================================================================

La sincronización se hace en executeBresenhamStep().

Variables:
- dominantStepCounter
- currentBlock.dominantSteps
- currentBlock.absSteps[]
- bresenhamError[]

Proceso:
1. Se suma absSteps[i] al error del eje.
2. Si el error supera dominantSteps, ese eje da STEP.
3. Se resta dominantSteps al error.
4. Se agrega bit correspondiente al stepMask.
5. Se manda stepMask al PIO.
6. Se actualiza posición lógica.
7. Se incrementa dominantStepCounter.

Esto reparte los pasos de ejes secundarios de forma proporcional al eje dominante.

18. PREDICCIÓN / PLANEACIÓN DE MOVIMIENTO
================================================================================

En este template, la palabra predicción debe entenderse como la capacidad de
preparar matemáticamente un movimiento antes de ejecutarlo físicamente.

La predicción no es por encoder ni por sensor. Es una predicción de trayectoria,
pasos, distancia, velocidad y aceleración basada en la posición actual y el
destino solicitado.

Existen dos niveles:

1. Movimiento directo con moveLinearTo()

Cuando se usa:

motion.moveLinearTo({100.0f, 50.0f, 10.0f}, 40.0f);

el controlador calcula inmediatamente:

- Posición actual de cada eje.
- Posición destino de cada eje.
- Diferencia en milímetros.
- Diferencia en pasos.
- Dirección de cada eje.
- Cantidad absoluta de pasos por eje.
- Eje dominante.
- Distancia geométrica total.
- Velocidad real permitida.
- Aceleración real permitida.
- Perfil trapezoidal.
- Proporción de pasos de cada eje.

Este camino no usa MotionPlanner. Es directo.

2. Movimiento planeado con queueLinearTo() + prepareQueuedMoves()

Cuando se usa:

motion.queueLinearTo(...);
motion.queueLinearTo(...);
motion.prepareQueuedMoves();

el sistema primero guarda las intenciones de movimiento como RawLinearMove.
Después MotionPlanner analiza esos puntos, detecta esquinas y produce
PlannedLinearMove. Finalmente AdvanceMotionController convierte cada
PlannedLinearMove en MotionBlock para ejecutarlo físicamente.

Flujo:

queueLinearTo()
    Guarda el movimiento crudo como RawLinearMove.

prepareQueuedMoves()
    Analiza la secuencia de puntos.
    Detecta esquinas.
    Calcula si se puede aplicar suavizado.
    Genera segmentos intermedios si es necesario.
    Produce PlannedLinearMove.

update()
    Toma un PlannedLinearMove.
    Lo convierte en MotionBlock.
    Ejecuta físicamente el movimiento con Bresenham y PIO.

================================================================================
19. LINEALIDAD DE TRAYECTORIA
================================================================================

El sistema ejecuta movimientos lineales sincronizados.

Cuando se pide:

motion.moveLinearTo({100.0f, 50.0f, 10.0f}, 40.0f);

el controlador interpreta que debe moverse desde la posición actual hasta:

X = 100 mm
Y = 50 mm
Z = 10 mm

La linealidad no significa que todos los motores giren a la misma velocidad.
Significa que todos los ejes avanzan proporcionalmente para que el punto de la
máquina siga una línea recta en el espacio de coordenadas.

Ejemplo:

X necesita 8000 pasos.
Y necesita 4000 pasos.
Z necesita 1000 pasos.

Entonces:

- X es el eje dominante.
- Y debe dar la mitad de pasos que X.
- Z debe dar una octava parte de pasos que X.

El sistema reparte esos pasos durante el movimiento dominante para que todos
lleguen juntos al destino.

Archivos y elementos relacionados:

- MotionBlock.deltaMm[]
- MotionBlock.deltaSteps[]
- MotionBlock.absSteps[]
- MotionBlock.dominantSteps
- AdvanceMotionController.bresenhamError[]
- AdvanceMotionController.executeBresenhamStep()

================================================================================
20. LOOK-AHEAD / LOCK AHEAD / ADELANTO DE TRAYECTORIA
================================================================================

El usuario se refirió a "lock ahead". En el contexto de control de movimiento,
el concepto normalmente se conoce como "look-ahead".

En este template existe un look-ahead inicial o simplificado. No debe
confundirse con un look-ahead completo tipo GRBL, LinuxCNC o controladores
industriales.

Lo que SÍ tiene actualmente:

- MotionPlanner puede mirar varios movimientos antes de ejecutarlos.
- Puede comparar un segmento con el siguiente.
- Puede detectar cambios de dirección.
- Puede detectar esquinas.
- Puede calcular un punto antes de la esquina.
- Puede calcular un punto después de la esquina.
- Puede generar segmentos Bezier para suavizar.
- Puede asignar velocidades de entrada y salida básicas.
- Puede bajar velocidad en curvas con blendSpeedScale.

Lo que NO tiene todavía como look-ahead avanzado:

- Forward pass completo.
- Reverse pass completo.
- Junction deviation real tipo GRBL.
- Cálculo avanzado por jerk.
- Buffer planner dinámico con reoptimización continua.
- Feed hold.
- Replanificación en tiempo real durante ejecución.
- S-curve.
- Jerk control.
- Closed-loop por encoder desde RP2040.

Para Codex:
El template tiene un concepto base de look-ahead dentro de MotionPlanner.
Si se solicita look-ahead avanzado, se debe extender MotionPlanner sin romper la
separación de responsabilidades.

================================================================================
21. MOTION BLOCK
================================================================================

MotionBlock es una estructura central del sistema.

Archivo:

lib/objMotion/MotionBlock.h

MotionBlock representa un movimiento ya convertido y listo para ejecutarse.

No es solo un destino. Es una unidad ejecutable que contiene toda la información
necesaria para generar los pulsos físicos.

Contiene:

axisCount
    Cantidad de ejes involucrados.

deltaSteps[]
    Diferencia en pasos entre posición actual y destino.

absSteps[]
    Valor absoluto de pasos por eje.

direction[]
    Dirección de cada eje.

dominantSteps
    Mayor número de pasos entre todos los ejes.
    Define la duración discreta del movimiento.

deltaMm[]
    Diferencia en milímetros por eje.

distanceMm
    Distancia geométrica total del movimiento.

requestedSpeedMmS
    Velocidad solicitada por el usuario.

nominalSpeedMmS
    Velocidad realmente permitida después de validar límites.

entrySpeedMmS
    Velocidad de entrada al bloque.

exitSpeedMmS
    Velocidad de salida del bloque.

accelerationMmS2
    Aceleración permitida del bloque.

valid
    Indica si el bloque es válido.

El MotionBlock se crea principalmente en:

AdvanceMotionController::buildLinearBlockToArray()

Relación:

Usuario pide movimiento en mm.
AdvanceMotionController crea MotionBlock.
TrapezoidProfile calcula perfil.
executeBresenhamStep reparte pasos.
StepperPioPulseEngine manda máscara.
stepper.pio genera el pulso físico.

22. RAW LINEAR MOVE

RawLinearMove representa una intención de movimiento antes de ser procesada.

Archivo:

MotionPlanner.h

23. PLANNED LINEAR MOVE
================================================================================

PlannedLinearMove representa un movimiento ya procesado por MotionPlanner, pero
todavía no ejecutado físicamente.

Puede representar:

- Un movimiento lineal original.
- Un segmento de curva generado por suavizado.

Contiene:

axisCount
targetPositionMm[]
requestedSpeedMmS
entrySpeedMmS
exitSpeedMmS
isBlendSegment
valid

Flujo:

RawLinearMove
    pasa a MotionPlanner

MotionPlanner
    genera PlannedLinearMove

AdvanceMotionController
    convierte PlannedLinearMove en MotionBlock

MotionBlock
    se ejecuta con Bresenham, perfil trapezoidal y PIO

================================================================================
24. PERFIL TRAPEZOIDAL
================================================================================

El perfil trapezoidal evita que los motores arranquen o se detengan de golpe.

Archivos:

TrapezoidProfile.h
TrapezoidProfile.cpp

Calcula:

- Pasos de aceleración.
- Pasos de crucero.
- Pasos de desaceleración.
- Velocidad máxima alcanzable.
- Si el perfil es trapezoidal o triangular.
- Distancia de aceleración.
- Distancia de crucero.
- Distancia de desaceleración.

Perfil trapezoidal:

Acelera -> mantiene velocidad -> desacelera

Perfil triangular:

Acelera -> desacelera

El perfil se calcula al iniciar un MotionBlock:

currentProfile = TrapezoidProfile::calculate(...)

Luego calculateCurrentStepDelayUs() usa ese perfil para calcular el tiempo entre
eventos STEP.

================================================================================
25. SINCRONIZACIÓN DE EJES
================================================================================

La sincronización de ejes NO la hace MotionPlanner.

La hace:

AdvanceMotionController

mediante:

- buildLinearBlockToArray()
- prepareBresenham()
- executeBresenhamStep()

Variables importantes:

- dominantStepCounter
- bresenhamError[]
- currentBlock.dominantSteps
- currentBlock.absSteps[]
- currentBlock.direction[]

Ejemplo:

X = 8000 pasos
Y = 4000 pasos
Z = 1000 pasos

dominantSteps = 8000

Durante esos 8000 eventos:

- X da 8000 pasos.
- Y da 4000 pasos distribuidos.
- Z da 1000 pasos distribuidos.

Así todos finalizan al mismo tiempo.

Aunque el usuario pida una velocidad alta, el controlador limita la velocidad

AdvanceMotionController::limitSpeedByAxisLimits()

Esto evita que un eje supere su velocidad máxima individual.

AdvanceMotionController::limitAccelerationByAxisLimits()

Esto evita que un eje supere su aceleración máxima individual durante un
movimiento combinado.

28. LIMITACIÓN POR PIO / FRECUENCIA STEP

minimumStepIntervalUs

El controlador usa:

limitSpeedByPulseEngine()

29. SUAVIZADO DE TRAYECTORIA
================================================================================

El suavizado se configura con:

motion.configurePathSmoothing(
    enabled,
    blendDistanceMm,
    blendSegmentLengthMm,
    maxBlendSegments,
    blendSpeedScale
);

Parámetros:

enabled
    Activa o desactiva suavizado.

blendDistanceMm
    Distancia máxima para recortar esquina.

blendSegmentLengthMm
    Longitud aproximada de cada segmento de curva.

maxBlendSegments
    Máximo de segmentos por curva.

blendSpeedScale
    Factor de velocidad en curvas.

El suavizado solo aplica cuando:

- Se usa queueLinearTo().
- Se llama prepareQueuedMoves().
- Hay al menos dos movimientos.
- Hay al menos dos ejes.
- blendEnabled es true.

Con un solo eje no se genera suavizado geométrico.

30. DIFERENCIA ENTRE moveLinearTo Y queueLinearTo
================================================================================

moveLinearTo()
--------------
Movimiento directo.

No usa MotionPlanner.
No usa suavizado geométrico.
Convierte inmediatamente a MotionBlock.
Se ejecuta con update().
Es ideal para mover a un solo punto.

queueLinearTo() + prepareQueuedMoves()
--------------------------------------
Movimiento planeado.

Usa MotionPlanner.
Permite analizar varios puntos.
Permite suavizar esquinas.
Genera PlannedLinearMove.
Luego cada PlannedLinearMove se convierte a MotionBlock.
Es ideal para rutas con varios puntos.
