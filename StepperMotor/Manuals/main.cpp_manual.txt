MANUAL DETALLADO Y ACTUALIZADO: StepperMotor/prgProject/main.cpp
=================================================================

Objetivo del archivo
--------------------
`main.cpp` es el punto de entrada de la aplicación de ejemplo para Raspberry Pi Pico/Pico SDK. La versión actual ya no define manualmente las configuraciones de ejes dentro del código principal; ahora carga una receta de máquina desde una microSD y, si esa carga falla o la receta es inválida, usa una receta embebida por defecto como respaldo seguro de demostración. Después crea el motor de pulsos PIO, construye dinámicamente los objetos de eje sobre memoria estática, crea el controlador de movimiento, ejecuta HOME, realiza movimientos lineales y prepara una cola de movimientos con suavizado de trayectoria.

Dependencias incluidas
----------------------
- `pico/stdlib.h`: funciones base del Pico SDK (`stdio_init_all`, `sleep_ms`, `tight_loop_contents`, tipos `uint`, etc.).
- `<new>`: habilita placement new para construir objetos `AdvanceStepperAxis` dentro de un buffer estático preasignado.
- `<stdio.h>`: habilita `printf` para mensajes por consola/USB serial.
- `StepperAxisConfig.h`: estructura de configuración eléctrica, cinemática, límites y HOME de cada eje.
- `AdvanceStepperAxis.h`: clase que representa un eje físico y convierte posiciones entre mm y pasos.
- `StepperPioPulseEngine.h`: clase que genera pulsos STEP usando PIO.
- `AdvanceMotionController.h`: controlador que coordina varios ejes, planifica y ejecuta movimientos.
- `HomeSequenceFunctionBlock.h`: bloque funcional que valida, configura y ejecuta la secuencia HOME.
- `MachineRecipe.h`: estructuras `MachineRecipe`, `MachineRecipeAxis` y parser de recetas.
- `SdRecipeStorage.h`: acceso a microSD y ruta por defecto `DEFAULT_RECIPE_PATH`.
- `DefaultRecipe.h`: texto de receta embebida `DEFAULT_MACHINE_RECIPE_TEXT` usado como fallback.

Objetos principales instanciados
--------------------------------
1. `MachineRecipe recipe`: contenedor de configuración parseada: versión, ejes, cantidad de ejes y orden HOME.
2. `SdRecipeStorage storage`: lector de microSD configurado con `SdSpiConfig::createDefault()`.
3. `MachineRecipeParser parser`: interpreta el texto de receta y rellena `MachineRecipe`.
4. `StepperPioPulseEngine pulseEngine`: genera pulsos STEP para hasta 8 pines consecutivos desde GPIO 2.
5. `axisStorage`: matriz de bytes alineada para alojar objetos `AdvanceStepperAxis` sin usar heap dinámico tradicional.
6. `AdvanceStepperAxis* axes[]`: arreglo de punteros a los ejes construidos.
7. `AdvanceMotionController motion`: controlador multieje que recibe los punteros de ejes, la cantidad real de ejes y el motor PIO.

Flujo general de ejecución
--------------------------
1. Inicializa la consola serial y espera 2 segundos.
2. Carga la receta desde `0:/config/axes.txt` en microSD.
3. Si no puede leer microSD, o si el texto leído no parsea correctamente, intenta usar la receta embebida.
4. Si tampoco puede parsear la receta embebida, queda detenido en un bucle infinito seguro.
5. Configura el generador PIO de pulsos STEP.
6. Construye un objeto `AdvanceStepperAxis` por cada eje descrito en la receta.
7. Crea el `AdvanceMotionController`.
8. Ejecuta HOME según el orden especificado en la receta; si no hay orden, usa el orden por defecto del controlador.
9. Mueve a una posición segura posterior al HOME.
10. Ejecuta un movimiento lineal directo.
11. Habilita suavizado de trayectoria y prepara una cola de tres movimientos lineales.
12. Entra en un bucle infinito llamando `motion.update()` para mantener vivo el planificador/ejecutor.

Métodos y funciones definidos o usados
--------------------------------------
- `loadRecipeFromMicroSdOrFallback(MachineRecipe& recipe)`: función estática local que llena `recipe` desde microSD o desde receta embebida.
- `executeRecipeHomeSequence(AdvanceMotionController& motion, const MachineRecipe& recipe)`: adapta el arreglo `homeOrder` de la receta a la interfaz de `executeHomeSequence`, que acepta `std::initializer_list`.
- `stdio_init_all()`: inicializa la E/S estándar del Pico SDK.
- `sleep_ms(2000)`: pausa el arranque para permitir abrir el monitor serial.
- `SdSpiConfig::createDefault()`: devuelve la configuración SPI por defecto para la microSD.
- `storage.begin()`: inicializa el subsistema de microSD.
- `storage.readTextFile(...)`: lee el archivo de receta a un buffer de texto.
- `storage.end()`: libera/cierra el acceso a microSD.
- `parser.parse(...)`: valida y transforma texto de receta en `MachineRecipe`.
- `parser.getLastError()` y `storage.getLastError()`: devuelven diagnóstico textual del último error.
- `executeHomeSequence(...)`: bloque funcional de HOME por nombres de ejes.
- `motion.init()`: inicializa hardware/controlador cuando se usa el HOME por defecto.
- `motion.homeAll()`: ejecuta HOME de todos los ejes.
- `motion.moveToSafePositionAfterHome(...)`: movimiento seguro posterior a HOME.
- `motion.moveLinearTo(...)`: movimiento lineal directo a una posición objetivo.
- `motion.configurePathSmoothing(...)`: configura mezcla/suavizado entre segmentos.
- `motion.clearMotionQueue()`: limpia la cola de movimientos.
- `motion.queueLinearTo(...)`: añade movimientos lineales a la cola.
- `motion.prepareQueuedMoves()`: planifica y valida la cola antes de ejecutarla.
- `motion.isBusy()` y `motion.update()`: consultan/avanzan la ejecución de movimiento.
- `motion.emergencyStop()`: detención de emergencia ante error de movimiento o planificación.
- `tight_loop_contents()`: función del Pico SDK para bucles intencionalmente vacíos.

Notas de diseño y recomendaciones
---------------------------------
- La actualización principal respecto al manual anterior es que la configuración de ejes dejó de estar codificada directamente en `main.cpp`; ahora se obtiene de `MachineRecipe` mediante microSD o fallback embebido.
- El uso de placement new sobre `axisStorage` evita fragmentación o dependencia del heap, una práctica conveniente en firmware.
- El arreglo `axes` tiene capacidad máxima fija, pero solo se construyen `recipe.axisCount` elementos.
- Las posiciones de ejemplo tienen tres coordenadas. Si se usan recetas con más o menos de tres ejes, conviene revisar que las llamadas con listas de 3 valores sean compatibles con la validación del controlador.
- `executeRecipeHomeSequence(motion, recipe)` devuelve un código, pero actualmente `main` no lo valida. Si HOME falla, el programa podría seguir intentando movimientos; se recomienda capturar el retorno y detener en caso de error.
- Las llamadas a `queueLinearTo` no comprueban su retorno individual en este archivo. La validación final se realiza con `prepareQueuedMoves()`, pero si la API devuelve errores útiles por inserción, sería recomendable registrarlos.
